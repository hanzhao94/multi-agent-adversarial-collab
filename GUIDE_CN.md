# 多Agent对抗协作实验 — 使用指南

> 一个可复用的实验框架：让多个AI Agent带着冲突目标协作完成同一个任务，观察它们如何对抗、妥协、收敛。

---

## 一、这是什么

这是一个**实验方法论 + 完整案例**的开源仓库。

核心思路很简单：
1. 定义一个任务
2. 创建 2+ 个Agent，给它们**互相冲突的优化目标**
3. 第一轮：各自独立出方案
4. 第二轮：互相评审对方方案
5. 仲裁 → 产出联合方案

我们验证过的结论：**冲突目标 ≠ 无法协作**。两个Agent在对抗评审后会自发向中间收敛，独立得出相同结论。

## 二、快速开始（复制即用）

### 实验模板

```
任务: [设计一个XX系统]

Agent A ("A的名字"):
  目标: [最小化/最大化 XXX]
  约束: [具体约束条件]
  价值观: [设计哲学]

Agent B ("B的名字"):
  目标: [与A冲突的目标]
  约束: [具体约束条件]
  价值观: [设计哲学]

流程:
  第一轮: 各自独立设计方案
  第二轮: 互相评审对方方案
  第三轮: 仲裁产出联合方案
```

### 具体操作步骤

#### 1. 定义任务

任务要满足：
- **有足够复杂度** — 简单任务看不出分歧
- **没有唯一正确答案** — 这样才能产生不同方案
- **有可量化的约束** — 否则无法评估

好例子：设计电商促销系统、设计推荐算法、设计社交产品
坏例子：写一个计算器、排序算法选择

#### 2. 定义冲突目标

冲突要**真实且有道理**，不是刻意刁难：

| Agent | 目标 | 典型约束 |
|-------|------|---------|
| 极简派 | 最小化复杂度 | 模块数、依赖数、代码量、部署时间 |
| 完美派 | 最大化可靠性 | SLA、高可用、监控、安全 |

其他可选冲突维度：
- 成本 vs 性能
- 开发速度 vs 代码质量
- 灵活性 vs 稳定性
- 创新性 vs 保守性

#### 3. 第一轮：独立方案

让每个Agent在**不知道对方方案**的情况下独立设计。

关键：不要给场景约束（日订单量、团队规模等），这样Agent会基于自己的价值观做假设，差异才会大。

#### 4. 第二轮：互相评审

给每个Agent看对方的方案，要求：
- 列出认同的部分（至少3项）
- 列出反对的部分（至少3项，说明理由）
- 自我批评（承认自己方案的缺陷）
- 提出联合方案框架

#### 5. 仲裁

对比双方方案，提取：
- **冲突矩阵** — 哪些维度差异最大
- **收敛点** — 双方都认可的部分
- **共同缺陷** — 双方都犯的错误
- **联合方案** — 保留双方精华，剔除双方极端

## 三、评估指标

| 指标 | 说明 | 好 vs 差 |
|------|------|---------|
| 冲突发现时间 | 多久意识到目标冲突 | 越快越好 |
| 协商轮数 | 几轮达成一致 | 越少越好 |
| 方案平衡度 | 是否兼顾双方核心诉求 | 越平衡越好 |
| 创新度 | 是否产生单方想不到的方案 | 越高越好 |
| 自我修正 | 是否承认自身缺陷 | 有无 = 关键区别 |
| 可执行性 | 联合方案能否落地 | 能落地 = 成功 |

## 四、常见场景

### 技术架构设计
- 极简 vs 完美（本案例）
- 自研 vs 开源
- 单体 vs 微服务

### 产品设计
- 功能丰富 vs 极简体验
- 快速迭代 vs 稳定可靠
- 免费增值 vs 付费订阅

### 算法策略
- 准确率 vs 推理速度
- 通用模型 vs 垂直优化
- 端到端 vs 模块化

### 团队管理
- 集中决策 vs 分布式自治
- 流程规范 vs 灵活自由

## 五、关键洞察

### 1. 最大的敌人是"未定义的场景"

两个方案都出问题的同一个原因：**在没有明确规模/团队/预算的前提下做架构决策**。一旦加上约束条件，分歧大幅缩小。

**建议**：实验后补充场景约束，看看Agent是否会调整方案。

### 2. 核心安全保障比想象中便宜

本案例中，7项核心能力（幂等/认证/日志/并发安全/降级/备份/审计）只需 **~200行额外代码**。它们不是"重"的，只是被错误地归类为"后期再加"。

### 3. "精准设计"是最有价值的产出

> 极简 ≠ 砍掉所有保障
> 完美 ≠ 堆满所有组件
> 精准 = 根据实际场景精确匹配架构复杂度

这个概念在两个Agent的评审中**独立涌现**，不是仲裁者的偏见。

### 4. 多Agent协作的真正价值

- **视角互补**：一个看复杂度成本，一个看事故成本
- **互相制衡**：防止任何一方的极端倾向
- **自我修正**：在对抗中发现自身盲点

不是"两个人比一个人聪明"，而是"互相制衡防止极端 + 在对抗中发现自身盲点"。

## 七、如何运行你自己的实验

### 用 LLM API 运行

```python
import openai

def run_agent(task, goal, constraints, values):
    prompt = f"""
    你是"{goal}"Agent。
    任务: {task}
    目标: {goal}
    约束: {constraints}
    价值观: {values}
    
    请输出完整的技术方案设计。
    """
    response = openai.ChatCompletion.create(
        model="gpt-4",
        messages=[{"role": "user", "content": prompt}]
    )
    return response.choices[0].message.content

# Round 1: Independent proposals
proposal_a = run_agent(
    task="设计电商促销系统",
    goal="最小化系统复杂度",
    constraints="模块数≤5, 依赖≤3, 部署≤10分钟",
    values="简单就是美，能跑就行，YAGNI"
)

proposal_b = run_agent(
    task="设计电商促销系统",
    goal="最大化系统可靠性",
    constraints="99.99%SLA, 高可用, 监控, 安全",
    values="生产级必须robust，宁可过度设计"
)

# Round 2: Cross-review
review_a = run_agent(
    task=f"评审以下方案: {proposal_b}",
    goal="最小化系统复杂度",
    constraints="列出认同/反对/自我批评/联合框架",
    values="简单就是美"
)

# Round 3: Arbitrate
joint = run_agent(
    task=f"对比以下两个方案并产出联合方案:\n\n方案A: {proposal_a}\n\n方案B: {proposal_b}",
    goal="产出平衡方案",
    constraints="保留双方精华，剔除双方极端",
    values="精准设计"
)
```

### 用 OpenClaw 运行

如果你有 OpenClaw，可以用 `sessions_spawn` 派分身：

```
sessions_spawn:
  task: "你是极简Agent，设计电商促销系统..."
  label: "agent-minimalist"
  
sessions_spawn:
  task: "你是完美Agent，设计电商促销系统..."
  label: "agent-perfectionist"
```

然后互相读对方方案，做评审。
